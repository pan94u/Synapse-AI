> 本文档完整记录了 Synapse AI 从构想到成型的全部讨论过程，包括每一轮对话的输入、分析、决策和迭代。

## 第一轮：初始计划

### 用户输入
要求实现一个基于 Anthropic Skill 规范的智能 Agent 系统，支持 Mac 桌面应用和 Web 应用。

### 初始设计
- **定位**：个人工具级 Agent（"Super Agent"）
- **技术栈**：Bun + Hono + Next.js + OpenAI SDK + Playwright + Turborepo
- **AI 引擎**：MiniMax 2.5 + Claude API（均走 OpenAI 兼容格式）
- **核心能力**：对话、工具调用、Skill 系统、记忆、知识库、浏览器自动化
- **项目结构**：Monorepo，packages/ 下分 shared、agent-core、server、web、knowledge
- **实现分 6 个 Phase**：基础框架 → 工具系统 → Skill 系统 → 记忆知识库 → 浏览器 → Web UI

### 决策点
- 选 Bun 而非 Node：原生 TS 支持，速度快
- 选 Hono 而非 Express：轻量高性能，TS 优先
- 用 openai npm 包统一对接 MiniMax 和 Claude：两者都兼容 OpenAI 格式，一套 SDK 搞定
- MiniMax 的 `<think>` 标签需要单独解析

## 第三轮：加入 MCP Hub

### 用户输入
"我们还需要规划一件事情，我们需要一个 mcp 中心，来帮我接入我们的整体数字化系统，你觉得呢，毕竟我们的是在企业中"

### 分析过程
用户揭示了关键背景：**这不是个人工具，而是企业级平台**。需要与企业数字化系统集成。MCP (Model Context Protocol) 是 Anthropic 提出的标准协议，正好适合做企业系统集成层。

### 设计决策

**新增 `packages/mcp-hub/`：**
- hub.ts — MCP Hub 主类
- client.ts — 与单个 MCP Server 通信 (JSON-RPC over stdio/SSE)
- registry.ts — Server 注册表：发现、注册、注销
- lifecycle.ts — 生命周期管理：启动、停止、重启、健康检查
- auth.ts — 鉴权网关：统一管理企业系统凭证
- aggregator.ts — 工具聚合器：合并所有 Server 的 tools
- router.ts — 请求路由：将 tool call 路由到正确的 Server
- health.ts — 健康监控：心跳检测、自动重连、降级策略
- audit.ts — 审计日志
- rate-limiter.ts — 限流器

**新增 `packages/mcp-servers/`：**
- 初始包含：database, http-api, feishu, dingtalk, wechat-work, git, notion

**MCP 协议选择：**
- 遵循 Anthropic MCP 规范（JSON-RPC 2.0）
- 支持 stdio（本地子进程）和 SSE（远程服务）两种传输
- MCP 三大能力：Tools（可调用操作）、Resources（可读数据）、Prompts（提示词模板）

**关键架构原则：**
- "Skill 定义怎么做，MCP 定义连什么"
- 配置支持环境变量引用，敏感凭证永远不硬编码
- 基于 `@modelcontextprotocol/sdk` 官方 SDK 构建

## 第四轮：企业系统对接修正

### 用户输入
"我们不用钉钉，但是我们人财法 crm erp 等等数字化系统"

### 分析过程
- 移除钉钉 MCP Server（不用的系统不放进来）
- 用户明确了核心业务系统：人事(HR)、财务(Finance)、法务(Legal)、CRM、ERP
- 需要重新组织 MCP Server 的分层架构

### 设计决策

**MCP Server 重新分层为三层：**

| 层 | 说明 | Servers |
|---|------|---------|
| 基础设施层 | 通用连接器 | database, http-api, git, email |
| 协作通讯层 | 协作平台 | feishu, wechat-work |
| 业务系统层 | 人财法+CRM+ERP | hrm, finance, legal, crm, erp, bi, dms |

**引入适配器模式（Adapter Pattern）：**

核心洞察：同一个业务领域（如人事），不同企业可能用不同产品（用友HCM/北森/SAP）。不应该为每个产品做一个 MCP Server，而是：
- 每个业务领域一个 MCP Server（标准化接口）
- 内部通过 Adapter 对接不同厂商
- 换系统只需改 adapter 配置，上层 Agent 和 Skill 零改动

```
HRM MCP Server
├── 标准接口: hrm_get_employee(), hrm_query_attendance()
└── Adapter Layer
    ├── 用友HCM Adapter
    ├── 北森 Adapter
    └── SAP HCM Adapter
```

**每个业务 Server 的标准化工具定义：**
- HRM: hrm_get_employee, hrm_list_org, hrm_query_attendance, hrm_get_payroll, hrm_list_candidates
- Finance: fin_query_ledger, fin_list_invoices, fin_get_expense, fin_query_budget, fin_ap_ar_summary
- Legal: legal_list_contracts, legal_get_contract, legal_create_sign, legal_compliance_check, legal_list_cases
- CRM: crm_list_customers, crm_get_opportunity, crm_update_lead, crm_sales_pipeline, crm_service_tickets
- ERP: erp_query_inventory, erp_list_orders, erp_get_bom, erp_procurement_status, erp_production_plan

**新增数据与文档层：**
- bi — BI/报表系统
- dms — 文档管理系统

## 第五轮：定义愿景

### 用户输入
"现在我们说下我们的愿景吧，这个超级智能体是用来作为我们创造 ai 原生性组织的基石，让所有员工可以从基础的繁重工作解脱出来，让组织告别繁重的流程做到高合规快速聚焦相应业务创造价值，让企业具有跨时代的竞争力"

### 分析过程
用户的愿景揭示了更深的目标 — 不是做一个"效率工具"，而是要**重新定义组织运作方式**。

对愿景进行了拆解：
1. **解放人** — 从繁重事务性工作中解脱
2. **消灭流程摩擦** — AI 就是流程本身
3. **高合规零等待** — 合规内嵌而非外挂
4. **聚焦价值创造** — 知识不在人脑里，新人秒变老兵
5. **跨时代竞争力** — Agent 主动发现问题和机会

### 差距分析（愿景 → 架构映射）

| 愿景目标 | 需要的架构能力 | 当前状态 |
|----------|--------------|---------|
| 解放人 | 角色画像系统（Personas） | 缺失 |
| 消灭流程摩擦 | 流程引擎 + 合规内嵌 | 缺失 |
| 高合规零等待 | 合规引擎（Pre-Hook） | 缺失 |
| 知识不丢 | 组织记忆（非个人记忆） | 缺失 |
| 主动智能 | 主动智能引擎 | 缺失 |

### 设计决策

**新增 4 个 packages：**

1. **`packages/personas/`** — 角色画像系统
   - 每个角色有专属 AI 画像：职责、权限、默认技能、沟通风格、知识范围
   - YAML 配置文件定义角色（config/personas/*.yaml）
   - 登录时自动注入角色上下文到 Agent system prompt
   - 内置 7 个角色模板：CEO、HR经理、财务总监、法务、销售、运营、工程师

2. **`packages/compliance/`** — 合规引擎（初始设计为 Pre-Hook）
   - 规则 DSL（YAML 格式）
   - 在 tool_call 到达 MCP Server 前拦截
   - 四种结果：allow / deny / require_approval / modify

3. **`packages/proactive/`** — 主动智能引擎
   - 三种模式：定时任务（cron）、事件触发、阈值监控
   - YAML 配置文件定义主动任务

4. **`packages/org-memory/`** — 组织记忆
   - 与个人记忆分离
   - 四类知识：制度规章、决策记录、经验教训、最佳实践
   - 按角色权限控制访问

**AI 原生组织 vs 传统组织对比表：**
梳理了 7 个维度的对比（信息流转、决策方式、战略推进、流程执行、知识管理、跨部门协作、合规检查），让愿景变得具体可感。

## 第六轮：整体架构图

### 用户输入
"挺好了，但是你整体看下，我们的愿景很清楚了，细节也不错了，但是我们缺乏一个整体架构图来承接我们的愿景，包含我们的细节之间的关系"

### 分析过程
用户指出了一个关键缺失：有愿景（why）、有细节（how），但缺少中间的**整体架构**（what）来把两者连接起来。

### 设计决策

**9 层分层架构（从用户到底层系统）：**

```
① 用户与角色层     — 谁在用（Personas）
② 交互体验层       — 看到什么（Web UI）
③ 服务网关层       — 怎么连（Hono API）
④ 智能中枢层       — 大脑（Agent 引擎）
⑤ 合规引擎         — 守门人（Pre-Hook + Post-Hook）
⑥ 主动智能引擎     — 主动行动
⑥.5 决策智能引擎   — 数据→洞察→决策→战略
⑦ 能力层           — 技能+工具+记忆+知识
⑧ 企业集成层       — MCP Hub
⑨ 企业数字化系统   — 实际的业务系统
```

**5 条核心数据流：**
- 流程 A：用户主动对话（被动模式）— 用户提问 → 角色识别 → Agent 规划 → Pre-Hook → MCP 执行 → Post-Hook → 返回
- 流程 B：Agent 主动行动（主动模式）— 定时触发 → 执行 Skill → 推送结果
- 流程 C：Pre-Hook 拦截 + 审批 + Post-Hook 通知 — 不合规操作被拦截 → 审批通过后继续
- 流程 D：组织记忆闭环 — 对话中发现有价值的信息 → 沉淀到组织记忆 → 新人入职时推送
- 流程 E：决策智能闭环 — 数据采集 → 指标计算 → 洞察发现 → 决策建议 → 追踪复盘

**9×9 模块关系矩阵：**
展示了角色画像、合规引擎、主动智能、决策智能、Agent 引擎、MCP Hub、Skill 系统、组织记忆、个人记忆之间的依赖和协作关系。

**一句话架构总结：**
> 角色决定了你是谁，Agent 理解你要什么，合规 Pre-Hook 确保能不能做，MCP 连通去哪做，Skill 定义怎么做，合规 Post-Hook 确保做得对不对，决策智能把数据变成洞察、把洞察变成决策、把决策变成可追踪的战略，主动智能让 Agent 不等你说就做，组织记忆保证做过的事和做过的决策永远不会忘。

### 用户输入
"帮我给我们的平台起一个名字吧，我们一起共创下，合规引擎应该可以 pre hook 和 post-run hook"

- 选项 D：组合造词（如 OrgOS、FlowMind）

用户选择了 B（简洁英文）

**第二轮问题：具体形式**
- 选项 A：Synapse — 突触，神经元之间传递信号的连接点
- 选项 B：Cortex — 大脑皮层，负责高级认知功能
- 选项 C：Nexus — 连接点，核心枢纽
- 选项 D：Pulse — 脉搏，生命活力的象征

用户选择了 A — **Synapse AI**

**命名理由：**
> Synapse /ˈsɪnæps/ — 突触，神经元之间传递信号的连接点。
> 像突触连接神经元一样，Synapse AI 连通组织的每个人、每个系统、每个流程。

### 合规引擎重大升级

**用户的关键洞察**：合规不只是事前拦截（Pre-Hook），还需要事后校验（Post-Run Hook）。

**之前的设计**：只有 Pre-Hook（执行前校验）

**升级后的双阶段架构：**

**Pre-Hook（执行前）— "这个操作能不能做？"**
- 角色权限校验
- 业务规则校验
- 时间窗口校验（如财务结算期禁止修改账目）
- 频率检查（防止异常重复操作）
- 前置条件检查（合同签署前必须法务审核）
- 参数修正（自动收窄超权限的查询范围）
- 结果：allow / deny / require_approval / modify

**Post-Run Hook（执行后）— "做完的结果合不合规？"**
- 敏感数据脱敏（身份证/银行卡/手机号自动遮掩）
- 数据分级标记
- 结果合理性检测（库存为负数 → 标记异常）
- 写操作影响范围确认
- 审计记账
- 联动触发（大额付款完成 → 通知财务总监）
- 回滚补偿（重复付款 → 自动冲销）
- 结果：pass / mask / flag / notify / revoke

**新增文件：**
- pre-hook.ts — Pre-Hook 执行器
- post-hook.ts — Post-Run Hook 执行器
- masker.ts — 数据脱敏器
- compensator.ts — 补偿/回滚器

**规则 DSL 扩展**：每条规则增加 `phase: pre | post` 字段

## 第八轮：决策智能引擎

### 用户输入
"还有一个事情，企业智能化和数字化都有一个共同的目的，就是如何通过数据的运营持续支持决策者和管理者进行管理决策完成对企业战略做出有效决策和持续推进，这个内容缺失了"

### 分析过程
用户指出了一个深层缺失：系统已经能连数据（MCP）、看数据（BI），但缺少从**数据到决策到战略闭环**的能力。

**BI ≠ 决策智能：**
| | 传统 BI | 决策智能 |
|---|--------|---------|
| 数据获取 | 人工拉报表 | Agent 持续采集 |
| 分析方式 | 看图表靠经验 | AI 自动发现趋势/异常/归因 |
| 输出 | 静态报表 | 洞察 + 决策建议 + 风险评估 |
| 战略关联 | 看数据和看战略是两件事 | 指标 ↔ 战略目标实时穿透 |
| 闭环 | 没有 | 决策→执行→效果→复盘全链路 |

### 设计决策

**新增 `packages/decision-engine/`：**
- collector.ts — 数据采集器（定期从 MCP 各系统拉取经营数据）
- metrics.ts — 指标体系（KPI 树定义、计算、同环比、基线）
- insight.ts — 洞察引擎（趋势/异常/归因/预测/关联/对标）
- advisor.ts — 决策顾问（方案生成、风险评估、历史类比）
- strategy.ts — 战略追踪（OKR → KPI → 数据穿透对齐）
- journal.ts — 决策日志（背景→选项→决策→结果→复盘）
- report.ts — 智能报告（自动生成分析报告）

**五层决策智能栈：**
```
战略层 — 战略目标 → OKR → 关键结果 → 衡量指标
  ↕ 对齐
指标层 — 经营指标体系（KPI 树）
  ↕ 计算
数据层 — 从 MCP 各系统持续采集
  ↕ 分析
洞察层 — AI 发现趋势、异常、归因、预测
  ↕ 建议
决策层 — 方案生成 + 风险评估 + 历史类比 → 决策日志 → 复盘 → 组织记忆
```

**6 种洞察类型：**
1. 趋势发现 — 时序回归分析
2. 异常检测 — 基于历史基线偏差
3. 归因分析 — 多维下钻 + 关联
4. 预测推演 — 趋势外推 + 情景模拟
5. 关联发现 — 跨系统数据关联
6. 对标比较 — 同环比 + 历史同期

**决策日志 → 组织记忆闭环：**
决策复盘后，经验教训自动沉淀到 org-memory。未来遇到类似场景时，Agent 自动引用历史决策。

**新增数据流 E：**
数据采集 → 指标计算 → 洞察引擎 → 主动预警 → 决策支持 → 决策记录 → 追踪复盘 → 组织记忆

## 第十轮：管控模式修正（重大转折）

### 用户输入
同样的推演要求，但**关键一词之差**：
"集团对各子版块是**投资管控型**"（而非运营管控型）

### 分析：为什么这是根本性的变化

**投资管控型 vs 运营管控型是完全不同的管理范式：**

| 维度 | 运营管控型 | 投资管控型 |
|------|-----------|-----------|
| 集团角色 | 超级运营总部 | 智慧资本方 + 战略引导者 |
| 子公司关系 | 执行单元 | 独立经营主体 |
| 集团管什么 | 日常运营过程 | 投资回报、组合优化 |
| 推广方式 | 自上而下强推 | 价值驱动，子公司自愿 |
| 核心场景 | 合规管控（强制执行） | 投资组合智能（数据驱动决策） |
| 成功标志 | 全覆盖率 | 自愿接入率 >80% |

### 设计决策

**完全重写 STRATEGY-5Y.md**，核心逻辑从"管住"变为"吸引"：

**三层价值模型：**
```
Layer 3：产业生态价值 — 从内部工具到产业基础设施
Layer 2：组合投资智能 — 投资组合洞察/协同发现/投退决策（集团独特价值）
Layer 1：单体企业效能 — 各子公司独立使用提升自身效率（基础）
```

**五年主线重构：**

| 年份 | 运营管控型主题 | 投资管控型主题 |
|------|--------------|--------------|
| 第一年 | 让集团"看见" | **种子与引力** — 不强推，做引力 |
| 第二年 | 从"看见"到"管住" | **引力扩散** — 口碑驱动，子公司主动要用 |
| 第三年 | 从"管住"到"赋能" | **价值涌现** — 网络效应，接入越多价值越大 |
| 第四年 | 从"赋能"到"重塑" | **智慧资本** — 从有钱的投资方到有智慧的资本方 |
| 第五年 | AI 原生组织 | **产业大脑** — 生态粘性，子公司被吸引而非被管住 |

**关键洞察：**

1. **推广策略完全不同**：不能发文件要求子公司使用。只能让子公司因为"真的有用"而主动拥抱。

2. **杀手级场景变了**：
   - 运营管控：合规管控（系统管行为）
   - 投资管控：投资组合智能（数据驱动投退决策）

3. **赋能含义变了**：
   - 运营管控：输出制度和流程，要求执行
   - 投资管控：输出能力和工具，你自己选择用不用

4. **初创企业的战略角色**：在投资管控型集团，初创企业是最好的"广告牌"，因为独角兽和子集团都在观望。

5. **薄软产业赋能的投资逻辑**：
   - 薄软产业管理水平低 → 利润被低效吃掉 → 提升管理 = 提升投资回报
   - Synapse AI 把赋能从成本项变成增值引擎
   - 这是集团投资薄软产业的 alpha（超额收益）来源

6. **终极护城河**：不是靠行政命令绑定子公司，而是靠价值绑定。子公司自愿留在生态里，因为"在这里比外面强"。

7. **五年后的一句话总结**：
   > 集团从"有钱的投资方"进化为"有智慧的产业大脑"。子公司不是被管住的，而是被吸引住的。

## 关键决策汇总

### 架构决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 运行时 | Bun | 原生 TS，快，兼做包管理 |
| 后端框架 | Hono | 轻量高性能，TS 优先 |
| 前端框架 | Next.js + shadcn/ui | 成熟生态，SSR + CSR |
| AI SDK | openai npm | MiniMax 和 Claude 统一接入 |
| MCP SDK | @modelcontextprotocol/sdk | Anthropic 官方，避免造轮子 |
| Monorepo | Turborepo | 管理多 package 构建 |
| 业务系统对接 | Adapter Pattern | 换厂商只改配置，上层零改动 |
| 合规架构 | Pre-Hook + Post-Hook 双阶段 | 事前管"能不能做"，事后管"做得对不对" |
| 记忆分层 | 个人记忆 + 组织记忆 | 个人来去，组织知识永不丢 |
| Skill 来源 | built-in / installed / custom | 内置不可删 / 市场安装 / 用户自创 |

### 产品决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 平台名称 | Synapse AI | 突触——连接组织的每个人、系统、流程 |
| 目标用户 | 全组织所有角色 | 不是给 IT 的工具，是给全员的 AI 伙伴 |
| 推广策略 | 价值驱动，自愿接入 | 投资管控型不能强推 |
| 先打哪个 | 初创企业 | 最轻、最快、最有示范效应 |
| 杀手级场景 | 投资组合智能 | 投资管控型集团的核心痛点 |
| 不用的系统 | 钉钉 | 用户明确不用 |
| 用的系统 | 人财法 CRM ERP + 飞书/企微 | 用户的实际数字化系统 |

### 战略决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 管控模式 | 投资管控型 | 用户修正，非运营管控 |
| 集团角色定位 | 智慧资本方 + 能力平台 | 不是管运营的，是投资+赋能的 |
| 赋能方式 | 提供选择，不要求执行 | 投资管控尊重子公司自治 |
| 成功指标 | 自愿接入率 >80% | 不是强推覆盖率 |
| 薄软产业策略 | AI赋能→管理提升→利润提升→投资回报 | 赋能从成本变增值 |
| 外部商业化 | 第四年开始 | 产业 SaaS + Skill 市场 + MCP 生态 |
